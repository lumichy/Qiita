少し前まで「これからはMCP（Model Context Protocol）がAgentの万能インターフェースになる！」と界隈が湧いていましたよね。皆さんも一度は「もうツール連携は全部MCPでいいんじゃないか」と思ったのではないでしょうか？

個人的にもMCPの登場テンションが上がって色々なServerを試したのですが、最近、海外の開発プラクティスを見ているとトレンドが静かに逆転し始めているのを感じます。

実は、PerplexityのCTOやY CombinatorのCEOといったトップ開発者たちが、内部ツールからMCPを外し、あえて原始的な「CLI（コマンドラインインターフェース）」へ回帰しているのです。最近爆発的に流行しているAgentプロジェクト「OpenClaw」を見ても、タスク実行はほとんどCLIコマンドや内部の関数に依存しており、MCPの出番はごくわずかです。

なぜ、モダンで高度なMCPではなく枯れた技術のCLIが再評価されているのか？本記事では、両者のメリット・デメリットを比較し、これからのAgentアーキテクチャの住み分けについて考察します。

## ざっくり比較：MCP vs CLI

| 項目 | MCP（Model Context Protocol） | CLI（Command Line Interface） |
|------|------|------|
| **設計思想** | アプリケーション間通信の標準化・JSON構造化伝達 | UNIX哲学に基づくコマンドのパイプライン結合 |
| **トークン消費** | 極めて大きい（仕様定義に大量のトークンが必要） | 極めて少ない（bashツール1つと少しのプロンプトで完結） |
| **実行効率** | 低い（一問一答の繰り返しで往復のラグが発生） | 高い（パイプと論理積で自律的に一括処理） |
| **安全性・権限** | 高い（事前に定義されたメソッドのみ実行可能） | 低い・危険（環境丸見えで、任意のコマンドが実行可能） |
| **主なユースケース** | 企業向け、クラウド共有、高セキュリティ要件 | 個人開発、ローカル環境、軽量Agent |


## なぜCLIに回帰するのか？CLIの2つの圧倒的勝因

### 1. トークン消費量が桁違いに少ない
MCPを本番導入しようとして、一番最初にぶち当たる壁がこれです。**「ツール定義のメタデータを全てプロンプトに乗せなければならない」**という重すぎる仕様。

例えば、GitHub向けのMCP Server。ここには44個のツールが含まれていますが、その説明ドキュメント（名前、説明、入力パラメータの形式など）だけでなんと約1683行、約6.3万文字にもなります。
これをトークンに換算すると**約14,268トークン**。APIコストにすると1回のリクエストを投げるだけで約0.3円が確定で飛びます。複数のMCPサーバーをスタックさせたら、さらにコストは跳ね上がります。

一方、CLIはどうでしょうか。
モデルに渡すツールは「`bash`を使えるよ」という定義ただ1つ。説明も十数行で済みます。トークン消費量は誤差レベルです。
ここ数ヶ月Agentを組んでいて実感しますが、Claude 3.5 SonnetやGPT-4oなどの最新LLMは、学習データとして膨大なCLIコマンド（`git`, `grep`, `gh`, `curl`…）をすでに“完全に”知っています。わざわざツールの使い方を教えなくても、勝手に適切な引数を選んで実行してくれるんです。マイナーな自作スクリプトを動かしたい時だけ、数行の手引書をプロンプトに添えれば十分でした。

### 2. 実行効率が圧倒的に高い（往復ラグがない）
MCPの動作モデルは、基本的に「LLMが考える → 一つのツールを呼び出す → 結果を受け取る → また考える → 次のツールを呼ぶ」というステップを踏みます。
このループ、実際に動かしてみるとわかるのですが、非常に待ち時間が長くてストレスが溜まります。

対照的に、CLIは**UNIXの偉大な遺産**をそのまま活かせます。
パイプ（`|`）や論理積（`&&`）を使って、LLMが一発で「長文コマンド」を生成してくれます。

```bash
# Agentが生成するCLIの例（一撃で複数タスクをこなす）
grep -rn "TODO" ./src | awk '{print $1}' > todos.txt && git commit -am "chore: extract todos" && git push
```

一度コマンドを発行すれば、あとはローカルのシェルが最後まで一気にやり遂げてくれます。LLMとの通信ラグが減るため、体感スピードは爆上がりします。実際、私の最近のスクリプトでも「とにかくbashに丸投げする」アプローチに変えてから、処理時間が半分以下になりました。

## MCPが捨てられない理由。不可代替な強みとは？

「じゃあもう全部CLIでいいじゃん！」と言いたいところですが、現場ではそうもいきません。MCPには、CLIでは絶対に担保できない強みがあります。

### 1. 実行パラメータが構造化されていて「壊れにくい」
CLIは文字列としてコマンドを組み立てる性質上、エッジケースに弱いです。
例えば、処理対象のファイル名に**シングルクォートやスペース**が含まれていたとします。LLMがうっかりエスケープ処理を忘れると、それだけで構文エラーになりタスクが機能不全に陥ります。

MCPの場合、引数は全てJSONのスキーマとして構造化されて渡されます。パラメータの境界が明確で、JSONパースの時点でエスケープも正しく行われるため、「コマンドラインの文法エラー」といったしょうもない理由でAgentがコケる確率を大幅に防げます。

### 2. クリティカルな環境での「安全性」が段違い
結局のところ、ここに行き着きます。
CLI環境（特にbashまるごと許可）をAIに与えるということは、**`rm -rf /`をいつでも実行できる権限をセットで渡す**ことと同義です。個人のPCでSandbox化されているならまだしも、企業のクラウド上で動く共有AgentにCLIを開放するのは、セキュリティ担当者が絶対に許さないでしょう。

MCPは「サンドボックス化された堅牢なAPI」として機能します。「ファイルの読み取りはできるが削除はできない」「DBのSELECTだけを許可する」といった具合に、あらかじめ開発者が実装・許可したメソッドしかAgentは叩けません。
暴走したAIがシステムを破壊するリスクを極限まで抑えることができるため、高い権限管理が求められるエンタープライズ領域では、引き続きMCPが主役です。

## まとめ：これからのAgentアーキテクチャの未来図

PerplexityやY Combinator周辺のトッププレイヤーたちがCLIに回帰したのは、彼らが「スピードとコスト効率、そしてLLMの推論能力の底上げ」に真っ先に気づき、最大限活用しているからです。

今後のAgentエコシステムは、技術の優劣ではなく**「用途による明確な住み分け」**が進むはずです。

*   **CLI（比重は急拡大）：**
    トークンの節約、圧倒的な実行スピード、低コストを武器に、個人開発者やローカルに常駐する軽量Agentの「大本命」として定着する。
*   **MCP（比重は縮小するが要として存続）：**
    JSONによる堅牢なパラメータ制御と高いセキュリティを活かし、安全性が最優先される企業向けサービス、またはクラウド共有環境での「最後の砦」になる。

もし今、あなたがAgentを開発していて、「MCPサーバーを繋ぎすぎてトークン消費ヤバい」「なんか動作が遅い」と悩んでいるなら、思い切って「ただbashを叩かせるだけ」のシンプルなCLI構成に先祖返りしてみてはいかがでしょうか？驚くほど身軽で、賢く動いてくれますよ！
